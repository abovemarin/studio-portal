# Session 7.1 — Security Audit Record (RBAC + Validation + XSS)

Verification audit of the security work built continuously across Modules 2–6. This is not
where security starts; it confirms that work held and enumerates the gaps. Each finding is
dispositioned as PASS, GAP, or NEEDS-TEST. The remediation decisions locked after review are
recorded at the end of this file.

---

## Checklist

### A. Route-handler RBAC — every /api handler (session? role? resource ownership?)

1. POST /api/auth/request-link — public by design; rate-limited and zod-validated.
2. GET /api/auth/callback — public; single-use token validity is the gate.
3. POST /api/auth/logout — session via Better Auth signOut.
4. GET /api/projects — session, then role branch; the client path is scoped by userId.
5. POST /api/projects — role: admin only.
6. GET /api/projects/:slug — session plus resource ownership (membership OR admin).
7. PATCH /api/projects/:slug — role: admin only.
8. DELETE /api/projects/:slug — role: admin only.
9. POST /api/projects/:slug/members — role: admin only (project and target user must exist).
10. DELETE /api/projects/:slug/members/:userId — role: admin only.
11. POST /api/projects/:slug/milestones — role: admin only (project must exist).
12. PATCH /api/milestones/:id — role: admin only (milestone must exist).
13. DELETE /api/milestones/:id — role: admin only.
14. POST /api/milestones/:id/comments — session plus ownership (membership via the milestone's project, OR admin).
15. DELETE /api/comments/:id — session plus ownership (comment author OR admin).
16. POST /api/milestones/:id/approve — session plus ownership (membership OR admin), plus a status guard.
17. app/admin/layout.tsx — server-side role gate for the whole admin subtree.
18. app/page.tsx and app/projects/[slug]/page.tsx — client identity plus membership gate.

### B. Data-layer read paths — every read in /lib/db

19. listProjectsForAdmin — admin-only caller; unscoped by design.
20. listProjectsForClient(userId) — innerJoin project_members plus where userId.
21. listProjectsForClientWithReview(userId) — the newest access path (6.3). It must gate identically
    to listProjectsForClient, and its milestones join must not surface another client's project data.
22. by-id / by-slug primitives (getProjectBySlug, getMilestoneById, getCommentById,
    getApprovalForMilestone, getProjectMember, getUserById) — intentionally unscoped; the calling
    route or page is responsible for the ownership check.
23. listMilestonesForProject, listCommentsForProject, listProjectMembers — scoped by projectId; the
    caller must already have gated access to that project.

### C. Input boundaries

24. Every route parses its params and/or body with a zod schema at the top of the handler; there are
    no hand-rolled presence checks.
25. The error envelope never leaks raw DB errors or stack traces; unexpected failures return a generic
    500 and are logged server-side.

### D. XSS — comment bodies sanitized on render

26. Client project detail (milestone-list.tsx) — the primary comment surface.
27. Any other surface where comments appear after the 6.3 changes (admin detail? dashboard?).

### E. Test beat — cross-client access denied (adversarial)

28. Client A (a member of project X but NOT project Y) reaching project Y and its milestones,
    comments, and approvals — by slug and by id, across every relevant route — must get a clean 404
    (not 403; no existence leak), per the 6.3 not-found decision.
29. Can client A's project list ever surface project Y's data through the review-count / last-activity
    join in listProjectsForClientWithReview?

## Findings

### PASS

- Admin-only routes (items 5, 7, 8, 9, 10, 11, 12, 13): each calls requireRole('admin') before any
  existence check, so a client is rejected with a uniform 403 whether or not the slug/id is real.
  Because the 403 is existence-independent, these are role gates, not existence leaks.
- Ownership routes (items 14, 16): each loads the milestone, then calls
  getProjectMember(milestone.projectId, user.id); a non-member non-admin is denied. Admin bypass is
  intentional and spec-correct (decision log 6.3b).
- DELETE /api/comments/:id (item 15): author-or-admin rule; the 403 for a non-author is a documented,
  deliberate choice. (See the remediation note about its residual, lower-priority existence leak.)
- GET /api/projects (item 4): the client branch calls listProjectsForClient(user.id), which is
  query-scoped, not a post-query filter.
- Page gates (items 17, 18): the admin layout redirects non-admins before any admin screen renders;
  the client pages call notFound() for a non-member (404, no existence leak); middleware checks only
  cookie presence and documents that it is not the security boundary.
- Data-layer scoping (items 20, 21): listProjectsForClientWithReview uses the SAME membership
  innerJoin plus where(userId) as listProjectsForClient. Its milestones LEFT join plus
  groupBy(projects.id) aggregate only the client's own already-scoped projects, so the review count
  and last-activity values cannot pull in another client's project. tests/db/projects.test.ts pins
  the membership clause byte-for-byte against listProjectsForClient via .toSQL().
- Input boundaries (item 24) and error mapping (item 25): all 16 routes zod-parse at the top; unknown
  errors funnel to a generic INTERNAL 500 with console.error server-side; slug collisions are
  pre-checked so a DB unique-violation never surfaces as a 500.
- XSS (items 26, 27): comments surface only on the client detail page, rendered as the JSX text child
  {comment.body} (React auto-escapes). The admin detail page and the dashboard render no comments.
- Magic-link and session hygiene: hashed single-use tokens, 15-minute TTL, invite-only sign-up;
  cookies are httpOnly, sameSite=lax, and secure in production; the role field is input:false so a
  client cannot self-elevate.

### GAP

- GAP-1 (Medium) — Existence leak: cross-client resources return 403, not 404, at the API layer.
  GET /api/projects/:slug returns 403 to a non-member on a project that EXISTS, versus 404 for an
  unknown slug — so a client can enumerate which project slugs are real. The same pattern applies to
  POST /api/milestones/:id/comments and POST /api/milestones/:id/approve (a real milestone in another
  client's project returns 403; an unknown one returns 404). This contradicts the 6.3
  no-existence-leak rule that the page layer already follows (projects/[slug]/page.tsx returns 404 for
  a non-member). The API predates 6.3 (Module 4); the page adopted the rule in Module 6, and the two
  diverged. The cross-client test beat requires 404 here. Resolution is Decision 1 below.

### NEEDS-TEST

- NEEDS-TEST-1 — End-to-end cross-client denial across the GAP-1 routes and the page gate, with REAL
  membership rows (not a mocked-null getProjectMember, which assumes the conclusion), driving client A
  against project Y by slug and by id and asserting the resolved 404 on each. Covered by Decision 4.
- NEEDS-TEST-2 — A live-DB proof that listProjectsForClientWithReview, called as client A, never
  contains project Y and never surfaces Y's inReviewCount or lastActivityAt. Only the compiled SQL
  shape is asserted today; this is the runtime data-level proof the .toSQL() test cannot give.
  Covered by Decision 4.
- NEEDS-TEST-3 (Low) — deliverableUrl (z.string().url()) accepts javascript: URLs, which
  milestone-list.tsx:68 renders as an <a href>. The input is admin-only (trusted), but a non-http(s)
  scheme in a clickable link is a latent XSS/redirect vector. Resolution is Decision 2 below.

### Notes in passing (out of 7.1 scope)

- Auth-after-validation ordering (Low): several routes zod-validate the body before requireRole /
  requireUser, so an unauthenticated caller can probe validation and receive a 400 before a 401/403.
  No data is touched pre-auth, so this is a minor ordering smell, not a vulnerability. Left as-is by
  decision (out of scope this pass).
- escapeHtml (lib/utils.ts) is exported and unit-tested but is never called in any render path; the
  actual XSS defense is JSX auto-escaping. Not a security gap, but the util and its test imply a
  mechanism that was never wired. Resolution is Decision 3 below.
- 7.2 preview (not touched here): the comment POST route has no rate limit; the in-memory limiter is
  single-instance only; secrets and .env.example hygiene are unchecked. All of this is Module 7.2.

## Locked remediation decisions (post-review)

1. GAP-1 → align to 404-no-leak on the three existence-leaking routes: GET /api/projects/:slug,
   POST /api/milestones/:id/comments, and POST /api/milestones/:id/approve. A non-member hitting a
   real resource in another project now receives the same 404 as a genuinely nonexistent one. Role
   gates stay 403/401: the admin-only routes (create/edit/delete) still return 403 to a client,
   because being told "you cannot do this" on an action whose resource you can legitimately see is not
   an existence leak. DELETE /api/comments/:id is deliberately left 403 in this pass (its author-or-
   admin rule operates on a resource visible to members; a full no-leak fix would require adding a
   project-membership gate it does not currently have — flagged as a follow-up, not silently changed).
   Each changed route's 403 test assertion is flipped to 404 in the same step as the route change.
2. URL scheme allow-list: restrict deliverableUrl to http/https at the zod boundary so javascript:
   and other schemes are rejected with 400 VALIDATION. Add a test asserting a javascript: URL is
   rejected and a normal https:// URL passes.
3. Remove the misleading escapeHtml artifact: delete the unused escapeHtml export from lib/utils.ts
   (after confirming no remaining callers), and rename the tests/api/comments.test.ts block from
   "read-time XSS neutralization" to name the real defense (JSX auto-escaping).
4. Write both cross-client tests this pass (NEEDS-TEST-1 and NEEDS-TEST-2). These require a live/
   in-memory test database; the current suite has no such harness (every API suite mocks lib/db, and
   tests/db/projects.test.ts states it has no live database). The harness approach is pending a user
   decision before implementation.

## Tracked follow-ups (opened by the 7.1 fixes pass)

The fixes pass landed Decisions 1–3 using the existing mocked-DB test pattern. These three items
are deliberately deferred so none is lost:

1. DELETE /api/comments/:id existence leak — accepted 403 this pass. A non-author non-admin gets 403
   for an existing comment (vs 404 for a nonexistent one), which reveals existence. Unlike the three
   GAP-1 routes, the proper fix is NOT a 403→404 flip: this route never loads the comment's project,
   so a correct no-leak rule (404 for a non-member of the comment's project, 403 for a member who is
   not the author) requires a NEW project-membership gate — new authorization logic, not an assertion
   change. Out of scope this pass; revisit as its own change. (Comment IDs are UUIDs, so practical
   enumeration is low-risk in the interim.)

2. Test-DB harness — to be built next pass. No live/in-memory test database exists today (every API
   suite mocks lib/db; tests/db/projects.test.ts states it has no live DB; CI runs vitest with no
   Postgres service). Decision pending: in-memory (e.g. pg-mem — new devDependency, driver-compat
   risk with drizzle/postgres-js) vs. real ephemeral Postgres (GitHub Actions `services:` block +
   a separate Vitest integration project that does not mock lib/db and applies migrations first — no
   new npm dependency). This harness is the prerequisite for the two tests below.

3. NEEDS-TEST-1 and NEEDS-TEST-2 — the deferred adversarial verification of THIS pass's GAP-1 fix.
   - NEEDS-TEST-1: integration test with real membership rows — client A (member of X, not Y) hits
     project Y and its milestones/comments/approvals by slug and by id across the GAP-1 routes and the
     page gate, asserting the resolved 404 on each. The 7.1 fixes-pass tests assert 404 only against a
     mocked not-a-member case; this is the real proof client A cannot enumerate project Y.
   - NEEDS-TEST-2: live-DB test seeding project Y under a different client, calling
     listProjectsForClientWithReview as A, asserting A's enriched list never contains Y and never
     surfaces Y's inReviewCount / lastActivityAt (the runtime data-level proof the .toSQL() shape test
     cannot give). Both depend on follow-up 2.
